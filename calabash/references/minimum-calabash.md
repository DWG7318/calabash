# Minimum Calabash

## Definition

Minimum Calabash is the original Calabash structure reduced to:

```text
Grandpa
→ Product Architecture
→ Ontology
```

It answers:

```text
Why does this product exist?
What real product is being built?
What canonical world does it contain?
```

## Grandpa

Grandpa is the product constitution: Owner intent, purpose, intended value, users,
non-negotiable judgments, hard boundaries, success direction, and Owner-exclusive
change authority.

## Product Architecture

Product Architecture describes roles, entry points, business/operational journey,
user and back-office surfaces, product responsibilities, data/evidence flow,
external capabilities, and outcomes.

It is not a module list, service map, or directory tree.

## Ontology

Ontology defines canonical concepts, names, relationships, ownership, states,
lifecycles, distinctions, and deprecated terms.

It is not merely a storage schema.

## Omitted Full Layers

Minimum Calabash does not include:

```text
Contract
Policy
Workflow
Action Catalog
Adapter
Eval & Audit
```

Omission means unresolved in Calabash, not unnecessary. Derive and freeze a needed
artifact before affected work is accepted.

## Creation

Minimum Calabash is not created by filling three blank pages. It is created through:

```text
prepared historical material
→ dynamic themes
→ filtered Owner decisions
→ immediate write-through
→ consistency audit
→ frozen version
```

The Theme Map may include questions that touch possible Full-layer upgrade even
when the current baseline remains Minimum.

## Upgrade

Upgrade when permissions, state transitions, external side effects, cross-system
contracts, agent adapters, audit, compliance, or semantic drift become material.

The upgrade is append-only and versioned.
